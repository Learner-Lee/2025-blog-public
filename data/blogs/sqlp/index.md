### 三层模块必须覆盖的知识点（最少这些，可多不可少）

**基础必学（目标：能看懂执行计划，能判断一条 SQL 慢在哪）**
- 数据库执行 SQL 的四个阶段：解析 → 重写 → 优化（生成计划）→ 执行
- `EXPLAIN` / `EXPLAIN ANALYZE` 是什么，怎么读它的关键字段（行数估算、扫描方式、JOIN 类型、cost）
- **全表扫描 vs 索引扫描**：什么时候哪个更快（不是索引永远赢！）
- B-tree 索引的工作原理（一句话：有序+多叉树，所以查找是 O(log n)）
- **最左前缀原则**：复合索引 `(a,b,c)` 为什么 `WHERE b=1` 用不上
- **覆盖索引**：为什么 `SELECT id, name` 比 `SELECT *` 在某些情况下快 10 倍
- **回表**是什么、为什么贵
- 常见慢的元凶清单：缺索引、`SELECT *`、`LIKE '%xxx'`、函数包裹列（`WHERE YEAR(created_at)=2025`）、隐式类型转换

**进阶（目标：能系统诊断中等复杂查询并给出优化方案）**
- **JOIN 算法**三种：Nested Loop / Hash Join / Merge Join，各自适用场景
- 子查询 vs JOIN vs CTE：什么时候哪个更快、为什么
- **分页深翻页问题**：`LIMIT 1000000, 20` 为什么慢，怎么改成游标分页
- **统计信息**（statistics）：优化器靠它估算行数，过期会导致计划走偏
- 索引设计权衡：索引不是越多越好（写放大、空间、维护成本）
- 慢查询日志：怎么开、怎么读、怎么排序找出"最该优化的那条"
- **SARGable** 概念：什么样的 WHERE 条件能用索引、什么样用不上
- ORDER BY / GROUP BY 的索引利用：能不能避免排序

**Master（目标：理解优化器的决策、并发与架构层面的影响）**
- **基于成本的优化器**（CBO）怎么算 cost：选择性、行数估算、IO/CPU 模型
- 优化器走偏的常见原因和对策（hint、改写、强制索引、更新统计信息）
- **隔离级别与锁**对性能的影响：MVCC 是什么、为什么 PostgreSQL 的"快照"和 MySQL 的"间隙锁"行为差很大
- 索引的进阶类型：哈希索引、GIN/GiST（PG）、全文索引、部分索引、表达式索引
- **分区表**：什么场景下真的有用，什么场景下是给自己挖坑
- 物化视图与查询缓存的取舍
- **架构级信号**：何时单机优化已到极限，必须考虑读写分离、分库分表、列存、OLAP 引擎
- 反模式集合：常见"网上抄来的优化技巧"在哪些场景下其实是负优化

> ### 1.4 索引不是越多越好 —— 写放大与低基数陷阱
> **🔧 这是什么**  
> 索引能加速读，但每次 INSERT / UPDATE / DELETE 都要同步维护所有相关索引。索引也占磁盘和内存。所以加索引是**用写性能和空间换读性能**的交易。
>
> **✅ 这么做，你会得到**  
> - 给**高基数 + 高频查询**的列加索引（如 `user_id`、`order_no`）：100 万行表上等值查询从 ~800ms 降到 ~2ms（约 400 倍）。
> - 不在低基数列上盲目加索引：避免索引文件臃肿和写放大。
>
> **⚠️ 改动它，会发生什么**  
> - 给 `gender`（只有 2-3 个值）单独建索引 → 优化器算下来"走索引再回表"还不如全表扫，索引基本不会被用，纯浪费。
> - 同一张表加 10+ 个索引 → 写入吞吐显著下降，UPDATE 一条记录要维护 10 个索引页。
> - 为了"以防万一"对每个 WHERE 用过的列都建索引 → 索引体积可能超过表本身，缓存命中率反而下降。
>
> **🧪 最小可验证示例**（PostgreSQL）
> ```sql
> -- 准备 100 万行测试数据
> CREATE TABLE orders (
>   id BIGSERIAL PRIMARY KEY,
>   user_id BIGINT,
>   gender CHAR(1),       -- 低基数：只有 'M'/'F'
>   amount NUMERIC(10,2),
>   created_at TIMESTAMP
> );
> INSERT INTO orders (user_id, gender, amount, created_at)
> SELECT (random()*100000)::BIGINT,
>        CASE WHEN random()<0.5 THEN 'M' ELSE 'F' END,
>        (random()*1000)::NUMERIC(10,2),
>        NOW() - (random()*365 || ' days')::INTERVAL
> FROM generate_series(1,1000000);
> ANALYZE orders;
> 
> -- 实验 1：高基数列加索引（应该有效）
> EXPLAIN ANALYZE SELECT * FROM orders WHERE user_id = 12345;
> CREATE INDEX idx_user ON orders(user_id);
> EXPLAIN ANALYZE SELECT * FROM orders WHERE user_id = 12345;
> -- 观察：Seq Scan 变成 Index Scan，耗时大幅下降
> 
> -- 实验 2：低基数列加索引（多半无效）
> CREATE INDEX idx_gender ON orders(gender);
> EXPLAIN ANALYZE SELECT * FROM orders WHERE gender = 'M';
> -- 观察：优化器多半仍然选 Seq Scan，因为 50% 的行都满足条件，
> -- 走索引反而更慢（要扫一半索引 + 一半回表）。
> ```
>
> **MySQL 差异**：`EXPLAIN ANALYZE` 在 MySQL 8.0+ 也支持但格式不同；统计信息更新命令是 `ANALYZE TABLE orders`。低基数索引在 MySQL InnoDB 上行为类似。
